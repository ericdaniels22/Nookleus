### Phase 2 — design iteration after first iPhone smoke

First install on Vanessa's iPhone showed two problems:

1. **No camera feed visible** — viewport was solid green where the camera should show. Root cause: Task 5 made the root `<div>` opaque green which sat OVER the native camera (rendered behind WebView with `toBack:true`). Fix in `20ea0e7`: removed root background, restructured middle band as flex-col with `bg-black` letterbox spacers wrapping a transparent `camera-preview-window` rectangle. Camera shows through.
2. **Vercel Toolbar overlay** appeared on the right edge of preview — that's a Vercel-feature attached to preview deployments, not our code, won't appear in TestFlight/prod.

Then Eric pivoted on the design itself:

- `68bc22c` **design swap: green bezels → black chrome + green action pills.** Eric: "I decided I don't like the green." All chrome (strips, letterbox bars, sheets, permission-denied) → `bg-black`. Pill action buttons (queue, Done, Back-to-job, Leave, Close, Continue) → `bg-[#0F6E56]` with white text. Icon-only top-cluster buttons stayed as white-on-black (no pill background — matches iOS camera convention). Shutter stayed white.
- `e83021c` shutter ring → green translucent (`border-[#0F6E56]/60`)
- `f807014` count moved from centered above shutter to upper-left of footer; font dropped `text-3xl` → `text-lg`
- `908952a` shutter ring thickened 3px → 5px
- `e04f5c7` top strip tightened — removed `pt-[max(env(safe-area-inset-top),16px)]` (was double-counting with Capacitor's `contentInset:'automatic'` which already offsets the WebView below the status bar); replaced with `pt-2 pb-2` so buttons sit right under the status bar
- `2c6ed68` camera viewport got an 8pt black side buffer (w-2 bars) + `rounded-3xl` corners. Native plugin's `startCamera` updated to render inside the same 8pt-inset rectangle.
- `a3aef21` **corner masks for visible rounded camera.** `rounded-3xl` alone did nothing because `camera-preview-window` is a transparent div with no fill — the native camera below renders as a hard rectangle regardless. Fix: 4 `radial-gradient` overlay divs at each corner that paint black L-shapes around the camera's hard corners, leaving a quarter-circle transparent area that creates the rounded visual.
- `b2d3e70` **camera flush under top strip + native↔CSS coord alignment.** Removed top letterbox spacer from middle band — camera now sits directly below buttons, all remaining space falls as a single bottom spacer. Also switched `startCamera` from hardcoded `topStripPt=90`/`bottomStripPt=150` constants (which didn't match the compacted top strip) to `getBoundingClientRect()` measurements of the actual rendered top + bottom strips. Native camera renders at exactly the same pixel rectangle as the CSS wrapper, so corner masks should always land on the camera.

## Decisions locked

(All explicitly confirmed by Eric during this session — overriding earlier 65b.2 grill decisions where applicable.)

- **Chrome color = black, NOT brand green.** Reverses the original 65b.2 plan's "brand-green bezel" decision. Brand green is now reserved for action pills only (Queue / Done / Back-to-job / Leave / Close / Continue) and the shutter ring + queue button.
- **Count = upper-left of footer, `text-lg` (18px).** Reverses the plan's "big number centered above shutter at `text-3xl`."
- **Shutter ring = 5px `#0F6E56` at 60% opacity.** Translucent green, thicker than the plan's white/40 at 3px.
- **Camera viewport sits flush below the top strip, NOT centered in the middle band.** All slack space accumulates as a single bottom spacer.
- **8pt black side buffer + `rounded-3xl` (24px) corner radius** on the camera viewport.
- **Top strip pt-2 / pb-2, no env(safe-area-inset-top) padding.** Capacitor's `contentInset:'automatic'` already offsets the WebView below the notch; adding env padding was double-counting and pushed buttons ~50pt too low.
