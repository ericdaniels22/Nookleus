## Pre-flight findings worth remembering

- **Statements are stored as HTML strings, NOT Tiptap JSON.** The Tiptap editor in `src/components/estimate-builder/statement-editor.tsx` outputs HTML; estimates / invoices store the HTML directly in `opening_statement` / `closing_statement`. The plan therefore ships a small **HTML→@react-pdf converter** (`src/lib/pdf-renderer/html-to-pdf.tsx` — ~80 lines, supports `<p>`, `<strong>`, `<em>`, `<ul>`, `<ol>`, `<li>`, strips images) instead of the Tiptap-JSON walker the spec originally implied. **Pattern**: when a spec says "Tiptap content", grep the editor file for `output:` / `value=` / `dangerouslySetInnerHTML` to confirm the actual stored shape before designing the converter.
- **Existing `/api/invoices/[id]/pdf` is a GET-only stub from 67b** (`src/app/api/invoices/[id]/pdf/route.ts`) using `generateInvoicePdf` from `src/lib/invoices/generate-invoice-pdf.tsx` and `src/components/invoices/invoice-pdf-document.tsx`. It also lacks a permission check (only `auth.getUser`) and writes timestamped attachments to `email-attachments/invoice-pdfs/{id}/{ts}.pdf` — diverges from the canonical "single overwrite at `pdfs/${orgId}/${jobNumber}/${docNumber}.pdf`" pattern. T17 deletes both files and rewrites the route as POST returning a 5-min signed URL with `view_invoices` permission. One existing caller in `src/components/invoices/invoice-read-only-client.tsx:45` is updated as part of the same task.
- **`requireAnyPermission` is the right gate for read routes that span estimates + invoices.** Item-library routes already use `requireAnyPermission(supabase, ["view_estimates", "view_invoices"])` — the same pattern lands on every `/api/pdf-presets` GET. Write routes use `requirePermission(supabase, "manage_pdf_presets")`.
- **Storage path convention:** `${orgId}/${jobNumber|contactId}/${id}.{ext}` — every existing path builder in `src/lib/storage/paths.ts` follows this shape (org prefix locked in 18a per `migrate-storage-paths.ts`). New helpers `estimatePdfPath` / `invoicePdfPath` follow suit. New private `pdfs` bucket (the `reports` bucket is public — wrong choice for customer documents).
- **Settings nav slot:** `src/lib/settings-nav.ts` is flat (no groupings). New "PDF Presets" entry slots after "Estimate Templates" so the catalog reads: Item Library → Estimate Templates → PDF Presets.
- **Pre-flight against `pg_get_functiondef` is required** for T21 (xactimate retire migration). The new `convert_estimate_to_invoice` body must carry over the existing implementation verbatim with two literal removals — and must NOT regress the I2 regex-safe due-days cast (67b cleanup) or the I4 inline totals recompute (also 67b cleanup). The plan's verification queries explicitly check both markers post-apply.

**Three patterns worth remembering** for the executor:

1. **`requireAnyPermission` for cross-feature read routes.** Any GET route that should be readable by users with EITHER `view_estimates` OR `view_invoices` uses `requireAnyPermission(supabase, ["view_estimates", "view_invoices"])`. Item-library does this; the new pdf-presets GET routes do too.
2. **Service client for storage uploads, RLS client for table reads.** The PDF render routes use the user's RLS-scoped `createServerSupabaseClient` to load the document + sections + line items (so RLS enforces tenant isolation), then switch to `createServiceClient` for the storage upload (because per-user grants on the bucket would be heavy). Same split as the existing `/api/contracts/[id]/regenerate-pdf` and `/api/invoices/[id]/pdf` (current GET) routes.
3. **5-minute signed URLs for download links.** The render routes return a `download_url` that's a 5-minute signed URL — long enough to browser-download, short enough that paste-bins / shared logs don't carry persistent access.
