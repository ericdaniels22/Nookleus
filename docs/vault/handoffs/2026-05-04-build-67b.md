## Pre-flight finding worth remembering

The plan called the polymorphism refactor "the meatiest, all entangled" and recommended it as one focused commit. That estimate was correct — but the actual landmine was that widening `onLineItemChange` and `onLineItemAdded` parameter types leaks into the **estimate** branch's setState (where the spread of `Partial<EstimateLineItem | InvoiceLineItem>` no longer satisfies `EstimateLineItem`). Initial tsc errored at the estimate branch; fix was to mirror the cast inside the estimate narrowing too (`partial as Partial<EstimateLineItem>` and `newItem as EstimateLineItem`), same shape as the invoice narrowing. **Pattern**: when widening a callback parameter from one member of a discriminated union to the union itself, every existing branch needs an explicit narrow back to its specific member — even the branches that weren't motivating the widening.

## Notes for next session

**67b is fully shipped.** No carry-over chips other than I1 (deferred to 67c). The cleanup pass closed every chip surfaced during 67b sessions 6–9.

**Three patterns worth remembering:**

1. **Widening a discriminated-union callback parameter requires symmetric narrows on all branches.** When `onLineItemChange(partial: Partial<EstimateLineItem>)` widens to `Partial<EstimateLineItem | InvoiceLineItem>`, the existing estimate branch's spread (`{ ...item, ...partial }`) stops satisfying `EstimateLineItem` because `partial.section_id` is now `string | null | undefined` (invoice's nullable type leaks in). Fix: cast `partial as Partial<EstimateLineItem>` inside the estimate narrowing, mirroring the invoice narrowing. The compiler can't narrow the union by the surrounding `if (prev.entity.kind === "estimate")` because the parameter is independent of state.

2. **`computeEstimateTotals` is already kind-agnostic — reuse it for invoice via output-key remap.** The helper signature only requires `subtotal / markup_* / discount_* / tax_rate` which all exist on both `Estimate` and `Invoice`. Output is `{ markup_amount, discount_amount, adjusted_subtotal, tax_amount, total }`. For invoice, destructure `total` and rename to `total_amount`; everything else is name-compatible. No new helper needed; just a thin `applyInvoiceTotals` adapter at the call site.

3. **`CREATE OR REPLACE FUNCTION` is reversible — applying via the Supabase MCP is low-risk relative to schema migrations.** Both 67b cleanup RPC swaps could be rolled back instantly by re-applying the original migration body. The actual risk is behavioral correctness of the new body, not "the migration could brick the schema." This made it tractable to apply directly via `apply_migration` rather than staging through Studio.
