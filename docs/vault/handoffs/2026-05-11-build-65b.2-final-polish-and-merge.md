### Phase 1 — fix the open coord-space mystery (1 commit, big diff)

The prior handoff left two visible bugs on iPhone after the `b2d3e70` "align native rect to CSS via DOM measure" attempt: bottom corners still square, and camera bleeding into the iOS status bar area. Diagnosed by reading the iOS plugin source directly:

- `node_modules/@capacitor-community/camera-preview/ios/Sources/CameraPreviewPlugin/CameraPreviewPlugin.swift:92-93` — the plugin divides incoming `x`/`y` by `UIScreen.main.scale` (so a passed `x=8` becomes a 2.67-point offset on a 3x device), but uses `width`/`height` as-is. **The asymmetric scaling is a plugin quirk, not a bug we'd fixed.**
- `CameraPreviewPlugin.swift:128` — `previewView` is added to `webView.superview` (= iOS root view, origin at screen top-left including the status-bar area). But our `getBoundingClientRect()` returns CSS coords relative to the WebView, which Capacitor's `contentInset:'automatic'` offsets below the status bar by `env(safe-area-inset-top)`. So passing `offsetY = topStripBottom` placed the native rect ~47pt above where the CSS wrapper actually rendered — top of camera leaked above the strip, bottom of camera ended ~47pt before the CSS-wrapper's corner masks (which is why the masks couldn't cover the actual bottom edge).

**The design pivot dropped the coord-fix problem entirely.** `7f96090` rewrote the layout: camera viewport is now full-width and **flush to the screen top** (iOS status bar overlays directly on the live preview), with only the **bottom** corners rounded. The native rect math collapses to `x=0, y=0` + `height = cssHeight + safeAreaTop` — both x and y are 0 so the plugin's `÷ UIScreen.main.scale` is a no-op, and the safe-area-top probe (a transient CSS div with `height: env(safe-area-inset-top)`) extends the native rect upward to cover the status bar area so the CSS-wrapper's bottom corner masks land exactly on the camera's hard bottom edge.

Side effects of the pivot: the X close + top-right cluster (flip/flash/mode/settings) became absolute overlays inside the camera viewport (white-on-image, no black-strip backdrop); bottom strip switched to `flex-1 justify-end` so the action row pins to the home indicator instead of floating at the top.

Smoke confirmed both bugs gone on Eric's iPhone after `7f96090` pushed and Vercel redeployed.

### Phase 3 — zoom-selector deferral

Eric asked about adding .5x / 1x / 3x zoom buttons (matching the reference image he supplied earlier in the session). Investigated and surfaced the constraints:

- `@capacitor-community/camera-preview` v8.0.1 exposes **no JS zoom method at all** — there's iOS pinch-handler code internally (`CameraController.swift:461-489`) but it's gated behind `enableZoom: true` + `UIPinchGestureRecognizer`. Not callable from JS buttons.
- The plugin's discovery session uses **`.builtInWideAngleCamera` only** (`CameraController.swift:49`). `.5x` ultra-wide and `3x` telephoto lenses are unreachable without code changes — `3x` would be digital zoom on the wide lens (quality loss), and `.5x` literally isn't possible without switching to `.builtInUltraWideCamera` / a virtual multi-cam device.

Three options laid out: pinch-only via flipping `enableZoom: true` (5 min, no native work); native patch via patch-package to add `setZoom()` + prefer `.builtInTripleCamera`/`.builtInDualWideCamera` (~1-2 hr, breaks the Vercel-preview iteration loop because Swift changes need Xcode rebuild, and lens availability varies per iPhone model); defer to a separate build. **Eric chose defer.** Filed in `docs/vault/builds/build-65b.2.md` "Future builds filed during grill" section (one detailed line item) so the next planning grill picks it up.

## Notes for the merge + next session

- **Vault-on-main vs code-on-PR pattern:** the prior handoff (`7886eaa`) was committed directly to `main`, while code stayed in the PR. This session diverges: handoff lands in the worktree branch and rides the PR merge into main. Either pattern works; the worktree-branch approach is more atomic ("code + handoff merge together") but requires the merge-from-main step done above to avoid `00-NOW.md` conflicts.
- **The `getBoundingClientRect` ↔ `CameraPreview.start` coord-space mismatch is fully documented in the `startCamera` callback comment** at `src/components/mobile/camera-view.tsx:75-90`. If a future plugin update changes the iOS scaling behavior (specifically the `÷ UIScreen.main.scale` on `x`/`y` at lines 92-93 of `CameraPreviewPlugin.swift`), revisit this — currently we sidestep the issue by passing `x=0, y=0`, but if we ever need a non-zero offset (e.g., for a smaller-than-screen preview window), we'd need to multiply by `window.devicePixelRatio` to compensate.
- **Subagent-driven development was not used this session.** Phase 1 was a single-agent diagnosis + rewrite (the layout pivot was load-bearing on the coord-space understanding, not parallelizable). Phase 2 polish was 6 quick controller-level edits where dispatching subagents would have been more overhead than savings. Phase 3 (zoom investigation) was research, not implementation.
- **The Vercel-preview iteration loop continues to work great** for web-only changes (just push → force-quit + relaunch). Any native plugin work (e.g., the deferred zoom build) will need to switch back to the Xcode rebuild loop.
- **Status bar text style on the live preview:** the iOS status bar overlays directly on the camera image now. We did NOT explicitly configure `statusBarStyle` (`.lightContent` vs `.darkContent`). The app uses whatever default Capacitor / the `@capacitor/status-bar` plugin sets. If the time/icons become hard to read against bright camera content, that's a separate fix (1-line `StatusBar.setStyle({ style: Style.Light })` from the camera mount effect).
