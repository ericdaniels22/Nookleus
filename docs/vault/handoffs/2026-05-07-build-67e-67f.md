## Notes for next session

- **The autosave fix at `22903f2` is the most consequential side change in this session, even though it's labeled as a "fix."** Before this fix, every Mark as Paid (or Mark Sent, or Void from the header — anywhere a status transition advances the row's `updated_at` while the user has the builder open) would have shown the user a confusing "Modified by another user — refresh to see changes" toast plus a stuck "Save failed — retrying" indicator. The bug existed since 67a's autosave concurrency was added, latent because Mark as Paid wasn't tested with a builder-open user session before. This fix benefits the entire estimate + invoice builder surface, not just 67e. The mechanism is generic: any external entity refresh (router.refresh after a status mutation, apply-template, anything that bumps `updated_at` server-side without going through the autosave PUT path) now re-baselines the autosave snapshot when no save is in flight. Worth keeping in mind for future builders that follow the same pattern.

- **The Void button on the invoice builder header had been hard-purging invoices since build 67d**, NOT soft-voiding them. The flow was: header-bar Void → `VoidDialog` → `onVoid(reason)` → `estimate-builder.tsx onVoid` → `fetch DELETE /api/invoices/{id}?reason=...`. Build 67d converted that DELETE endpoint to a hard-purge with audit log + storage purge. The optimistic UI update set `status: "voided"` so the user thought void worked, but server-side the row was destroyed. No one caught it because (a) it's a low-frequency action, (b) the optimistic UI hid the discrepancy until reload, (c) any reload would just see the row gone and treat that as "voided + trashed off the list" semantically. Now using `PUT /api/invoices/{id}/status` with `status: voided` instead — proper soft void with `voided_at` + `void_reason`, row preserved.

- **Pattern for `useAutoSave`-style hooks: external prop changes need explicit re-baseline.** When a server-driven mutation (status transition, apply-template, etc.) bumps the entity's `updated_at` outside the autosave path, the autosave hook needs to know. Two-prong design:
  1. Mount-init effect baselines once (already existed).
  2. Watch effect on `entity.updated_at` re-baselines when prop advances and no save is in flight (added this session).
  - Skip baseline if a save is in flight — that save will pick up the new `updated_at` from its own response.
  - Also dismiss any stuck stale-conflict toast on re-baseline so the user isn't left staring at a misleading message.
  - Same pattern likely applies to any builder/editor shell with optimistic concurrency.

- **`INVOICE_FIELDS` (and `ESTIMATE_FIELDS`) in `estimate-builder.tsx:54,84` includes `status`.** This means any status transition produces a non-empty diff in the autosave's PUT body, which triggers the autosave to fire. Good (status edits via direct field assignment would propagate); bad (status changed via the dedicated `/api/{entity}/[id]/status` route is now seen as a "pending dirty save" by autosave). The race surfaces this. The autosave fix handles it correctly by re-baselining BEFORE the watch effect's debounced timer fires. If a future build tightens the diff to "user-editable fields only" (excluding `status`), the same race wouldn't fire — but the re-baseline effect is still useful for the apply-template case (which can change many fields server-side).

- **The migration files at `supabase/migration-build67{e,f}-*.sql` are NOT applied via `apply_migration` — applied via `execute_sql`** to match Eric's preference for manual application. This means the Supabase migration registry doesn't track them; they exist in the repo for documentation + one-time-application only. Pattern matches `migration-build67d` and `migration-build67c1-*`. If `apply_migration` is ever wired into the CI/CD pipeline, the registry would need a backfill of these handoff-applied migrations.
