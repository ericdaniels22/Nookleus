## Notes for next session

- **Subagent-driven-development worked cleanly for this build.** Pattern: dispatch implementer subagent (haiku for mechanical tasks, sonnet for tasks involving design judgment or larger codebases), independently verify the resulting commit by reading the diff before dispatching reviewers, dispatch spec-compliance review, then code-quality review for non-trivial changes. Reviewer-found Important issues fed back to the implementer for amend. Skipped formal reviewer dispatches for 1-line/docs-only commits where the controller-level diff inspection was sufficient — saved cycles without sacrificing quality. **Two subagents flagged correct issues that would have shipped if blindly accepted:** the Task 1 vitest-config over-engineering and the Task 5 key-clobber race. The amend loop is genuinely earning its keep.

- **Plan-snippet APIs need verification, not blind paste.** Two of the plan's verbatim snippets had API mismatches against installed packages (`@capacitor/background-task` doesn't exist; `SecureStorage` is actually `SecureStoragePlugin`). Both implementers caught this — once via npm-not-found error, once via type-checker. Future executions should plan for this kind of friction; the recovery cost was small but unavoidable. **Lesson:** when a plan dictates exact import names or response shapes, the controller should pre-verify them against `npm view <pkg>` and the package's installed `.d.ts` before dispatching. This session's `@capawesome/capacitor-background-task` correction was easy because it was a typo at the package-name level; if it had been a typo at the API-surface level (different method names) it could have wedged Task 14 mid-execution.

- **The Supabase `mcp__claude_ai_Supabase__apply_migration` tool ran the migration cleanly against AAA prod with no friction.** Pre-flight `list_tables` and post-migration `execute_sql` verification + `get_advisors security` are the right ritual. Migration was forward-compatible (`uploaded_from text DEFAULT 'web'`) so the in-pass code change in Task 3 wasn't time-pressured — could have been deferred. Don't over-rely on this; some future migrations will be irreversible+code-coupled (e.g. column rename + backfill) and need tighter ordering.

The crypto-vault, exif-read, and upload-queue tests genuinely earn the dep — they catch math bugs (backoff intervals), API correctness (encrypt/decrypt roundtrip, IV randomness, race-free key cache), and fallback paths (non-JPEG blob → 0/0/1) that smoke alone wouldn't catch. **Don't expand vitest to cover Capacitor-bound modules** — those rely on real plugin behavior and are correctly smoke-only.
