### Track 3 — Inline fixes (commit `67e5506`)

**M1 — Resend error redaction.** Both `src/app/api/estimates/[id]/send/route.ts` and `src/app/api/invoices/[id]/send/route.ts` returned `e.message` raw at status 502 in the `sendOrgEmail` catch. Spec §11 Test 12 calls for "redacted provider error". Swapped both branches to `apiDbError(e.message, "POST /api/{estimates|invoices}/[id]/send dispatch", 502)` so the client now gets `{"error":"internal error"}` and the full Resend message lands in server logs (verified: `[api] POST /api/estimates/[id]/send dispatch: Resend error: …`). The `FromUnconfiguredError` literal token (`{error: "from_unconfigured"}`) is preserved because the modal keys on it.

**M2 — Preset Select label.** `src/components/send-modal/index.tsx`'s `<SelectValue>` was showing the raw UUID instead of the preset name. Root cause: base-ui `<Select.Value>` requires a function-as-children render prop to map the value to a label — `<SelectItem>` body text content is NOT auto-mapped (verified by reading `node_modules/@base-ui/react/esm/select/value/SelectValue.d.ts`). Added a render prop:

```tsx
<SelectValue placeholder="Select preset">
  {(v: string | null) => {
    if (!v) return null;
    const p = presets.find((x) => x.id === v);
    return p ? `${p.name}${p.is_default ? " (default)" : ""}` : v;
  }}
</SelectValue>
```

Verified: trigger now displays `"Estimate (default) (default)"` (the seed preset name happens to literally contain `"(default)"` plus the suffix marker — minor visual oddity; acceptable for now). Same Select wrapper is used in `item-form.tsx` and `add-item-dialog.tsx` with short string values where the bug is invisible — not fixing those proactively.

**Mn1 — Mid-session from_unconfigured token toast.** If admin clears `send_from_email` between modal-open (preview said OK) and Send click, the POST returns `400 {error: "from_unconfigured"}` and the modal was toasting the literal token to the user. Added a branch in the response handler that flips the modal to the from-unconfigured empty-state UI when it sees this code, instead of toasting. Verified end-to-end: opened modal with valid settings → SQL-zeroed `send_from_email` → clicked Send → modal flipped to "Configure your sending email first." with the settings link, no toast.

## Carry-over chips (deferred)

These came out of the code-reviewer pass but didn't warrant a fix-now:

### C1 — SendModal stays visually open after successful send (still open from prior session)

Base-ui Dialog flips `data-closed=""` on the popup but `display: grid` and `visibility: visible` persist (the close-out animation/unmount doesn't fire). State is correct (open=false, parent's useState flipped); user has to click Cancel/X to dismiss. Affects perceived UX of every successful send. Probably a base-ui Dialog vs `tailwindcss-animate` interaction quirk in this codebase. Not blocking; not destructive.

### C2 — Audit-row best-effort warn could mask a missed audit row in prod (Mn2)

`contract_events` insert is wrapped in `console.warn` on failure — invisible in prod logs without LogTail/similar. Suggestions: (a) include `event_type` + doc id in the warn line for grep-ability, (b) add an `audit_warning: true` field to the 200 response so the modal can show a non-fatal toast.

### C3 — Renderer 4xx errors masked as 500 by send routes (Mn3)

The renderer can throw `PdfRenderInputError` with a 400/404 status, but the send routes catch via `apiDbError` (always 500). Send routes pre-validate so this should never fire — pure defense-in-depth Nit; a `e instanceof PdfRenderInputError` short-circuit would let those leak through with their intended status if ever reached.

### C4 — Settings PATCH route is org-blind (Mn5, pre-existing build40+)

`src/app/api/settings/payment-email/route.ts:11-18, 51-59` — `getSettings()` does `.limit(1).maybeSingle()` with no `organization_id` filter. Pre-existing bug, but 67c2 widened the surface area by adding 4 new template fields (`estimate_send_*`, `invoice_send_*`). Recommend a separate cleanup chip — affects every multi-tenant org settings flow.

### C5 — `from_unconfigured` reply-shape inconsistency (N1)

Preview route returns `200 {from_unconfigured: true}`; send route returns `400 {error: "from_unconfigured"}`. Two callers, two shapes. Modal handles both, but worth unifying in a v2.

### C6 — Migration trigger missing `DROP TRIGGER IF EXISTS` (N2)

`supabase/migration-build67c2-send-via-email.sql:101-104` mirrors the 67c1 pattern. Per project memory, migrations are sequential + manual + not idempotent — but if the trigger is dropped + recreated by a future migration, the existing `OR REPLACE` on the function plus a fresh `CREATE TRIGGER` will collide. Add `DROP TRIGGER IF EXISTS … ;` before `CREATE TRIGGER` for forward safety. Same pattern would apply to 67c1's trigger if not already.

### C7 — Estimate route uses `BLOCKED_STATUSES` Set; invoice route uses inline `=== "voided"` (N3)

Functionally equivalent, 1-line divergence. Consolidate next time anyone touches either file.

### C8 — `MERGE_FIELDS` cross-domain coupling (N4)

T17 added `customer_first_name` to `src/lib/contracts/merge-fields.ts` for use in 67c2 email templates — but this same `MERGE_FIELDS` powers the contracts Tiptap pill picker. Verify the contracts merge-field picker either gets it intentionally or is filtered. Not a defect; cross-domain coupling worth noting before the next contracts change.

### C9 — F2 (Test 11 render-fail branch uncovered)

The route's pre-render preset-validation guard (line 90-96) catches `document_type !== "estimate"` and returns 400 `{"error":"invalid preset"}` BEFORE the renderer's try/catch can fire. So the `apiDbError` render-fail path (line 107-112) was not executed by Test 11. To exercise it directly would need a corruption that passes pre-validation but throws inside the renderer (e.g. a missing Storage bucket or service-role failure). Not blocking; render-fail branch coverage is from code-reading rather than execution.

### C10 — F4 (partial-unique-index pre-flight reminder)

`idx_pdf_presets_org_default WHERE is_default=true` is non-deferrable. Any future SQL that flips `document_type` while `is_default=true` collides if the target type already has a default. Combine with `is_default=false` in the same UPDATE, or demote the existing default first. Same lesson as the 67c1 cleanup pass.
