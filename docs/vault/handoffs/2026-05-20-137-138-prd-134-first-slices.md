## Notes for next session

**The post-merge `main` already carries the foundation for the remaining PRD #134 slices.** `CONTEXT.md` has the `Email account` / `Shared email account` / `Personal email account` terms; `docs/adr/0001-shared-and-personal-email-accounts.md` has the design rationale; `src/app/api/settings/users/[id]/route.ts` has the membership-guard pattern that slices #139 + #141 will mirror for the email-account-access module. Slice #139 should match the shape of `belongs-to-active-organization.ts` (the access-decision pattern from PRD #95 slice #97) — a pure module with a small per-case decision table, unit-tested in isolation, called from every consumer route.

**The #137 rollback was deliberately small.** It only undid the misbuilt parts and kept everything genuinely useful that was in the carry-over (membership guard + 404 test + `vi.fn` spies + `memberOrg` seed + the CONTEXT.md domain terms). Bundling the keepers with the rollback meant the slice landed as "the carry-over minus the misbuilt bits," not "rollback everything then re-add the keepers as separate work." This kept the diff legible — `git show 29457ab` shows precisely what was kept vs removed in one place.

**For #133 closure, the lesson generalises.** When a PRD breaks down into a single tracer-bullet slice via `/to-issues`, the slice PR's `Fixes` line will close the slice but not the parent PRD. Either reference both (`Fixes #133, Closes #135`) or close the parent manually after the slice merges. The current convention seems to be `Fixes <slice>` only — so retroactive parent-PRD closure is a routine cleanup task worth checking when scanning OPEN issues.

**Both PRs were truly orthogonal**, which made the parallel execution clean: #137 touched `src/app/api/settings/users/[id]/*` + `src/app/settings/users/page.tsx` + `CONTEXT.md` + `src/app/api/settings/users/__test-utils__/service-fake.ts`; #138 touched only `docs/adr/0001-shared-and-personal-email-accounts.md` (in a new directory). Zero shared paths, zero merge conflicts, zero rebase needed. For future "do two slices in parallel" requests, verify the file-path overlap is zero before launching — when slices share files, sequential is faster than parallel-with-conflicts.
