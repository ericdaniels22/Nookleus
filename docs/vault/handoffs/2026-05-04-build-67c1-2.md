## Plan deviations locked this session

These are the non-obvious places where the implementation diverges from the plan's literal text. **Future sessions should not revert any of these.**

1. **DROP TABLE prepended to T1 migration.** Build 67a's foundation migration shipped a placeholder `pdf_presets` table with the v1 ~28-column shape (0 rows, no app-code consumers). The 67c1 migration begins with `DROP TABLE IF EXISTS pdf_presets CASCADE` so the new slim 17-column shape applies cleanly. Eric authorized via "option 1" mid-session.

2. **`useToast` → `sonner`.** The plan's T7 + T8 imported `useToast` from `@/hooks/use-toast`. That hook does NOT exist in this codebase — `sonner` is the canonical toast library (see `src/components/contracts/contracts-section.tsx`, `src/components/compose-email.tsx`). All call sites converted to `import { toast } from "sonner"` + `toast.success(msg)` / `toast.error(msg)`. Pattern: any future plan code referencing `useToast` should be rewritten to `sonner` automatically.

3. **`InvoiceLineItem.amount` vs `EstimateLineItem.total`.** Plan T12's `sectionsTable` assumed both line-item types have `total`. They don't — `InvoiceLineItem` (line 724 of `src/lib/types.ts`) uses `amount` (with comment `// = total in estimate-land`). Fixed via discriminated read `"total" in i ? i.total : i.amount`, mirroring T13's identical pattern for the parent `total` vs `total_amount`. Same shape will be needed for any future code that polymorphically reads line-item totals.

4. **`render.ts` → `render.tsx`.** Plan named the dispatcher `render.ts` but the body is `renderToBuffer(<EstimatePdf {...input} />)` which is JSX. Renamed to `.tsx`. Imports use the path alias `@/lib/pdf-renderer/render` (extensionless), so all callsites resolve unchanged.

5. **T15 sample data field-list corrections.** Plan's `Invoice` literal was missing `po_number`, `memo`, `voided_by`, `qb_invoice_id` (all required `string | null` fields). Plan's `InvoiceLineItem` literal used `total: 250` (wrong field — type uses `amount`) and was missing `xactimate_code: null`. The handoff explicitly flagged this as worth eyeballing before dispatch — caught and fixed. The `xactimate_code: null` lines come out in T20's code cleanup commit.

6. **T16 + T17 `loadCompany` org-scoping + `logo_path` URL conversion.** Two real correctness bugs in the plan code:
   - `loadCompany` uses the **service** client, which bypasses RLS. Without `.eq("organization_id", orgId)` on the `company_settings` query, rows from all orgs would be merged into one mish-mash. Added the org filter to both routes' helpers.
   - `company_settings` stores the logo as `logo_path` (filename inside the public `company-assets` bucket), not `logo_url`. The plan reads `byKey.logo_url` which is always `undefined`. Fixed via `service.storage.from("company-assets").getPublicUrl(byKey.logo_path).data.publicUrl`. Pattern matches existing usage in `src/app/settings/company/page.tsx:57`.

7. **Hardening upgrades on T4 + T5 routes (reviewer-flagged Important findings).**
   - **Object-shape guard** on parsed JSON bodies (`if (!raw || typeof raw !== "object" || Array.isArray(raw))` → 400). The plan's `as PdfPresetCreatePayload`/`as PdfPresetUpdatePayload` cast doesn't narrow at runtime; a non-object body would TypeError on the first field access and surface as a redacted 500 instead of a clean 400.
   - **Boolean type validation** on every toggle field. T4's POST has an `asBool(v, default)` helper; T5's PUT has a `BOOL_FIELDS` array + loop returning `400 "<field> must be boolean"` for non-booleans. Without this, `{show_markup: "yes"}` would have hit Postgres and 5xx'd.
   - **`BoolKeys` constrained type** on T5: `type BoolKeys = { [K in keyof PdfPresetUpdatePayload]-?: PdfPresetUpdatePayload[K] extends boolean | undefined ? K : never }[keyof PdfPresetUpdatePayload]` — prevents the `BOOL_FIELDS` array from accidentally including non-boolean keys (e.g., `name`) at compile time.

8. **Lib-side 404 consistency.** `updatePreset` originally only pre-checked existence inside the `is_default === true` branch (the plan's literal). I lifted the pre-check to the top of the function so the route's `e.message === "preset not found"` → 404 mapping fires reliably regardless of whether the partial update flips `is_default`. Costs one extra single-row PK lookup per PUT; benefit is the route's 404 mapping is no longer a lie on the no-flip path.

9. **`html-to-pdf.tsx` whitespace-paragraph guard.** Reviewer caught that `<p>   </p>` (Tiptap can produce this when a user hits space + saves) would render a phantom 4pt-margin paragraph. Replaced `if (runs.length > 0)` with `if (runsHaveContent(runs))` checking trimmed content. Also flipped `import { JSX }` to `import type { JSX }` (only used in type position).

10. **Error-string contract documented.** `src/lib/pdf-presets.ts` header now documents the route-layer error-string contract: `"preset not found"` → 404, `"cannot delete default preset"` → 409. Future maintainers who edit those strings need to update every route handler that string-matches them. Reviewer also flagged sentinel error classes as the cleaner pattern; deferred until a third consumer site triggers the refactor (currently 1 lib file, 3 catch sites, 2 strings — too small to justify).

11. **Concurrency note in `updatePreset`.** Inline comment documents the last-write-wins model: row not locked, but the partial-unique index on `(organization_id, document_type) WHERE is_default = true` enforces "one default per (org, doc_type)" so concurrent set-as-default flips can't both succeed (the loser surfaces as a 5xx via apiError). Matches the 67a/67b model.
