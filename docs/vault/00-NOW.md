## Recently learned

- **The iPad install is a Capacitor TestFlight wrapper loading live Vercel content, NOT Safari Add-to-Home-Screen — and "in-app" detection has THREE shapes, not one.** Surfaced 2026-05-08 during 15h-followup ipad-pdf-access smoke when Eric reported the Download dead-end was still happening AFTER the first `display-mode: standalone` Share Sheet fallback shipped. Root cause: Eric's iPad install is the TestFlight Capacitor app (`capacitor.config.ts:9` sets `server.url = 'https://aaaplatform.vercel.app'` — thin WKWebView loading live Vercel content); inside Capacitor's WKWebView, `window.matchMedia("(display-mode: standalone)").matches` is **false** AND `window.navigator.standalone` is **undefined** — so the original detection silently fell through. Fix at `d52b04c` adds `window.Capacitor?.isNativePlatform?.() === true` to the OR. **Pattern:** any "is this in-app for share/download/install fallback purposes?" detection needs all three checks: `display-mode: standalone` (Safari PWA from Share menu) ∨ `navigator.standalone === true` (pre-spec iOS Safari Add-to-Home-Screen) ∨ `Capacitor.isNativePlatform() === true` (TestFlight / App Store builds). **Corollary:** web-only changes never require a TestFlight rebuild — Capacitor's live-bundle pattern means the iPad app picks up new code on next launch. TestFlight rebuilds are only needed when adding native plugins (`@capacitor/share`, `@capacitor/filesystem`, etc.) or modifying `ios/App` native code. iPadOS 16.4+ supports `navigator.share({ files })` inside WKWebView natively, so the plain web Share API path works without `@capacitor/share`. Memory at `~/.claude/projects/.../memory/project_ipad_install_capacitor.md`.

- **For files with unicode regex character classes OR escape-sequence text that needs to survive verbatim, the controller writes the file directly — don't delegate to subagents.** Surfaced 2026-05-08 during 15h-followup plan task 1 (`src/lib/contracts/pdf-filename.ts`). The file needed (a) the literal smart-quote codepoints `‐ ‑ ‒ – — ― ‘ ’ “ ”` (U+2010 through U+201D) inside the regex character classes AND (b) the literal escape-sequence text `\x00-\x7F` (NOT literal NUL/DEL bytes) inside `.replace(/[^\x00-\x7F]/g, "_")`. Two haiku subagent implementers in a row mangled it: first stripped smart quotes back to ASCII; second embedded literal NUL (0x00) and DEL (0x7F) bytes into the source so git flagged the file as binary ("Bin 0 -> 855 bytes"). The reviewer subagent was also confused by the same rendering issue and gave conflicting reports. Fix path: controller wrote the file directly via Write tool, then verified with `python3 -c "import re; src = open('...').read(); print('codepoints:', re.findall(r'[‐-”]', src)); print('null/del bytes:', '\\x00' in src, '\\x7f' in src); print('escape text:', '\\\\x00-\\\\x7F' in src)"`. **Pattern:** when correctness depends on specific unicode codepoints OR on the verbatim text of escape sequences as opposed to their byte values, write the file from the controller, not via a subagent dispatch. Subagents copy text through their context window in ways that can elide the distinction between "the source contains the four characters `\` `x` `0` `0`" and "the source contains the byte 0x00."

- **Next 16 server components disallow `dynamic({ ssr: false })` — wrap the dynamic import in a thin `"use client"` component.** Surfaced 2026-05-08 during 15h-followup plan task 3 (`src/app/contracts/[id]/view/page.tsx`). The plan said: `const SignedPdfViewer = dynamic(() => import("..."), { ssr: false });` directly in `page.tsx` (server component). Implementer correctly caught that this errors at build time and added `signed-pdf-viewer-client.tsx` (16 lines, `"use client"`) that holds the dynamic import — page imports the wrapper directly. Existing precedent in `contract-signer-view.tsx:12`. **Pattern:** any time a route page (server component) needs `dynamic({ ssr: false })` (e.g., react-pdf or any other module that crashes at module-eval under SSR), the import goes in a thin client wrapper, NOT directly in the page. The wrapper accepts the same props and renders the dynamic component. This is now the canonical pattern for the 15-series PDF surfaces (`signed-pdf-viewer-client.tsx`, `contract-signer-view.tsx`).

- **A new Tiptap node extension on one editor surface introduces a new HTML shape that other Tiptap editors silently strip.** Surfaced 2026-05-08 during the merge-field-pills session: introducing `MergeFieldNode` (atom inline node parsing `<span data-field-name="x">…</span>`) in the contract + payment template editors caused saved bodies to start containing pill spans. The other Tiptap consumers in the codebase — `send-contract-modal.tsx`, `compose-email.tsx`, `signatures/page.tsx`, `statement-editor.tsx` — use a plain `<TiptapEditor>` without the new extension. When those editors load a saved body with pill spans, Tiptap has no parseHTML rule for `data-field-name`, drops the span attributes, and keeps just the inner text node (which happens to be `{{x}}`). So functionally it still works (the inner bare-token survives + the resolver substitutes correctly + the bare-token signing-link guard at `api/contracts/send/route.ts:79` still matches), but UX is inconsistent: template editors show pills, the other editors show bare `{{token}}` text. **Pattern:** when adding a Tiptap node extension to one consumer, decide explicitly whether it should be threaded through every TiptapEditor caller (uniform UX) or scoped to the original consumer (accept inconsistency, document why). The default of "thread it through everything" is usually right when the saved HTML can flow between editors — which is true for any field that's stored once + edited from multiple surfaces. The default of "scope it" is right when the surfaces edit different content shapes (template configs vs. ad-hoc messages). Either way, document the choice — silent stripping looks like a bug to anyone debugging the receiving editor.
