## Recently learned

- **After a sweep-style task, the controller should grep to verify scope coverage independently — plans enumerate from memory and miss sites.** Surfaced during 67d Task 17 execution (2026-05-05 evening). The plan listed 4 modification sites for the active-list `.is("deleted_at", null)` filter sweep; the SDD subagent dispatched correctly stopped at those sites. A controller spot-check via `grep -n 'from("\(estimates\|invoices\)")' src/lib src/components src/app` found 5 additional aggregation sites — `job-detail.tsx` per-job invoiced widget, `accounting/margins.ts` per-job + global margin queries, `jarvis/tools.ts` per-job invoice list + cross-job outstanding-balance — all of which would have let trashed invoices inflate displayed totals as soon as the first invoice was trashed. Closed in follow-up commit `cbbf209`. **Pattern:** for any sweep that touches a class of call sites (active-list reads, mutating routes, audit inserts, etc.), the plan's enumerated list is "best memory at plan-write time" not "exhaustive." After the subagent reports DONE, run a parallel grep for the relevant pattern and verify each match was covered — the controller's grep is independent of the plan's memory. The subagent stopping at the listed sites is correct boundary discipline; the controller closing the gap is the matching half of the discipline.

- **A spec is a draft, not a contract — grep an existing call site before pasting any column or function name into a plan code block.** Surfaced during 67d plan-write (2026-05-05). The 67d spec §6 documented the audit-row JSON column as `payload`; the actual `contract_events` schema column is `metadata` (verified at `src/app/api/estimates/[id]/send/route.ts:172` during plan-write). The plan's audit-row INSERT blocks would have tsc'd fine but failed at runtime with "column 'payload' does not exist" on the first send-trash. Caught by copying the existing 67c2 send-route audit insert pattern into the new delete/restore/purge routes — the pattern carried the correct column name. **Pattern:** even when you wrote the spec yourself, treat its column/function/route names as a draft. Before pasting any name into a code block, grep an existing call site that uses the same shape — for `contract_events` audit rows, that's `grep -n 'from("contract_events").insert' src/`. Five seconds of grep beats five minutes of debugging.

- **Prepared SQL blocks need a 30-second pre-flight against constraints + FKs before running on prod.** Surfaced during 67c1 cleanup pass (2026-05-04 night). The handoff's prepared cleanup transaction had two ordering bugs that would each have failed at the first statement: (1) the partial-unique index `idx_pdf_presets_org_default WHERE is_default = true` is non-deferrable in Postgres, so promoting the old default while the new one is still `is_default=true` violates the index immediately; demote-then-promote is the correct order. (2) The FKs between `estimates ↔ invoices` (`converted_to_invoice_id` and `converted_from_estimate_id`) are NO ACTION (default), so deleting the linked invoice while the estimate's pointer is still set fails the FK; UPDATE the estimate first to clear the pointer, then DELETE the invoice. Both bugs surfaced from a `pg_indexes` + `pg_constraint` query that took maybe 30 seconds. **Pattern:** any prepared SQL block that touches multiple rows under a unique index or crosses an FK boundary deserves a pre-flight read of the index predicates + FK definitions before running, even when the block is "obviously safe." Default Postgres unique constraints are NOT deferrable; FK actions matter.

- **SQL-driven role flip is a clean test rig for client-side permission gating.** Used during 67c1 cleanup to verify the Test 11 UX-gap fix end-to-end without a separate test user. Sequence: (1) read `user_organizations.id` for the admin's membership in the target org; (2) UPDATE role to `crew_lead` + UPDATE the relevant `user_organization_permissions` row to `granted=false`; (3) reload the page in the live preview — auth-context `loadProfile` re-fetches profile + permissions on mount; (4) DOM-verify; (5) UPDATE role back to `admin` + UPDATE the permission row back to `granted=true`; (6) reload — admin state restored. Bounded restore window (single turn). Works against the live admin's own session — no separate user account needed. **Pattern:** when testing client-side permission gating, this is faster than provisioning a dedicated low-privilege test user and avoids invite-email flows. Restore is non-negotiable; do it in the same turn.

- **Permissions admin-bypass at `permissions-api.ts:44` is by design but counter-intuitive when testing.** `if (membership.role === "admin") return ok: true` short-circuits before the permission check. - **`useState(prop)` initializes once — server-driven prop advances need an explicit re-sync.** Surfaced during 67b Task 52 (Test 2 apply-template UI didn't update after success toast). Builder root pattern was `const [state, setState] = useState({ entity })` — when parent server-component re-fetched after `router.refresh()` and re-rendered with a new `entity` prop, the child's local state ignored it. The TS type system signs off because `supabase-js` types `.is()`'s second arg loosely. - **PostgREST `.contains(col, jsArray)` against a `jsonb` column emits invalid Postgres syntax.** supabase-js's `.contains()` serializes a JS array as a Postgres array literal `{x,y}`. That works against `text[]` columns but fails against `jsonb` columns with `"invalid input syntax for type json"` because PostgREST's `cs` operator on jsonb requires the value to be JSON-formatted (`["x","y"]`). Surfaced during [[build-67a]] post-ship browser verification — `damage_type_tags` and `section_tags` are `jsonb NOT NULL DEFAULT '[]'::jsonb` (per migration-build67a, lines 24-25), and AddItemDialog's library tab toasted the error when Eric clicked into the dialog with a damage_type chip pre-applied. - **Auto-save fetches need explicit timeouts.** A hung connection (mobile network drop, slow server) without `AbortController` makes React's saveStatus stick in `"saving"` forever; `inFlightRef` stays `true`, blocking subsequent debounced saves. The estimate builder's `useAutoSave` hook hit this in code review (commit `cb26a0b` fix) — added `FETCH_TIMEOUT_MS = 30_000` + `AbortController` to all four save paths (estimate-level PUT, line-item PUT, sections-reorder PUT, line-items-reorder PUT). AbortError is routed to the same error path as network errors. **General lesson:** any client-side fetch that sets a "saving" status flag needs a timeout-and-abort cycle; otherwise the flag is a lie under network failure.

- **Snapshot-based concurrency guards must bump the parent's `updated_at` even on no-op-looking child mutations.** The 67a I2 fix (commit `4a85c7e`) added server-side `updated_at_snapshot` enforcement to the sections + line-items reorder PUTs. - **PostgREST `.or()` accepts user input as comma-separated filter syntax — escape it.** The 67a Phase C sweep fixed three callers (`lib/item-library.ts`, `components/job-detail.tsx`, `api/email/contacts/route.ts`) where user-typed input was concatenated into an `.or(...)` filter expression. A search like `foo,bar` would be parsed as two filters with `bar` as a column-name fragment, which fails the request; worse, a value like `,deleted=is.true` could in principle escape into additional conditions (RLS contains the blast radius but the bug is real). Fix: new `lib/postgrest.ts` → `escapeOrFilterValue` wraps the value in PostgREST's documented quoted form (`"value"`) and escapes embedded `\` / `"`. `%` and `_` stay as ilike wildcards by design. **Canonical pattern:** any `.or(...)` or `.and(...)` whose value contains user input goes through the helper.

- **API routes should never return raw `error.message` on 5xx.** Postgres / Supabase errors leak schema details, constraint names, RLS policy text, and sometimes the offending row contents. Phase C added `lib/api-errors.ts` (`apiError` + `apiDbError`) that logs the full message server-side and returns `{error: "internal error"}` to the client. Wired into all 67a + item-library routes; the remaining ~80 routes still leak and are filed as a follow-up sweep. **Canonical pattern:** at every 5xx site, replace `NextResponse.json({ error: error.message }, { status: 500 })` with `apiDbError(error.message, "<METHOD> <route> <op>")`.
